The lambda NRVO trick

DISCLAIMER: All references to ABIs are accurate for the System V ABI on the x86 and x86_64 (a.k.a. amd64) architectures only. I am not familiar with other C/C++ ABIs.

PS: If the blocks of assembly are too intimidating to you even with the comments, don't worry about them. I explain what's happening later, in higher-level language.

First, some background: In the standard C/C++ ABI, if a function returns a value of a type that is too large to fit in a register, then an out-param is inserted before the normal parameter list, containg a pointer to caller-allocated memory (see page 22 of the linked pdf). That is, a function that looks like Huge foo(){ return make_huge_object(); } is really1 Huge* foo(void* dst){ Huge* obj = new(dst) Huge(make_huge_object()); return obj; } . The standard also mandates a kind of optimization called RVO (Return Value Optimization), where, given, say, std::vector<int> foo(){ return std::vector<int>{1,2,3}; } int main(){ std::vector<int> instance{foo()}; return 0; } , the temporary vector in foo must be constructed in-place into instance. It is not hard to see that under this ABI, the optimization is actually automatically implemented, as the address of instance is passed to foo's out-param, which would cause it to be constructed in-place.

But what if the value being returned is not a temporary? Specifically, what if we return a local variable? Well, in this case, the in-place construction optimization is called NRVO (Named Return Value Optimization), and is not mandated by the standard. It might not be immediately obvious why this optimization isn't guaranteed. In fact, for one (rather common) type of function, it is easy to perform NRVO. Specifically, functions that look like std::vector<int> foo(){ std::vector<int> local; do_something_with(local); return local; } can always be compiled as std::vector<int>* foo(void* dst){ std::vector<int>* local = new(dst) std::vector<int>(); do_something_with(*local); return local; } . To see why NRVO cannot be performed under all cases, consider std::vector<int> foo(bool condition){ std::vector<int> a, b; populate(a); populate(b); if(condition){ return a; }else{ return b; } } ; Here, although depending on condition, either a or b is returned, both vectors are unconditionally constructed. Clearly, the two vectors cannot go into the same location, so at least one vector must be constructed elsewhere and moved into the result location. (In fact, compilers might construct neither object in-place, though I'm not quite sure if it's just for simplicity or because "overwriting" the other value while returning has subtle edge cases that makes it not worth it.)

The interesting part is when we consider this function: void do_something_with(std::vector<int>&); std::vector<int> foo(bool condition){ if(condition){ return std::vector<int>{1,2,3}; }else{ std::vector<int> local{2,3,4}; do_something_with(local); return local; } } It seems that the NRVO here with local should be pretty easy to implement. The lifetimes of the temporary vector in the true branch of the if and the vector local are mutually exclusive, meaning only one vector is constructed for any invocation of foo (though it could be either one). Therefore, both vectors can be constructed in-place. GCC allocated a local std::vector on the stack, and then copied2 it into the return location (with the instructions between the XXX marks)! The move is cheap in this case, but still, we can definitely do better. Hey, I have an idea. How about... void do_something_with(std::vector<int>&); std::vector<int> foo(bool condition){ if(condition){ return std::vector<int>{1,2,3}; }else{ return []{ std::vector<int> local{2,3,4}; do_something_with(local); return local; // simple, unconditional NRVO case as shown above. }(); // Mandatory RVO from temporary. } } At this point, you might be yelling at me. "How could adding more lambdas help? You'd be lucky if the things get inlined properly without extra overhead!" but after that, you might start getting curious. After all, the return within the lambda does seem like the "simple" NRVO case we outlined above, and there's no doubt that returning the temporary returned by the lambda is a mandatory RVO case. But surely this can't work, GCC probably just gives up on the inner NRVO because it can't chain it with the outer RVO or something. The stack-allocated local is nowhere in sight. Stuffing the construction of local inside a lambda in the return statement caused the outer return in foo to upgrade to a forced RVO, and it seems that GCC can pick it up from here. It is truly weird that adding more indirection can make your code faster, but sometimes, that just might work.

1: Yes, the function also returns the exact same out-param that is passed in. This is required by the ABI, and is quite reasonable if you think about it, since rdi would've been clobbered by the call, and the caller probably wants to use the return value afterwards.

2: When I said "copy", I meant a copy of the underlying bytes, not of the C++ object. The code is in fact the result of inlining std::vector's move constructor, as you'd expect.
